# How to Use This Framework

A practical guide to get productive with Security Copilot for Purview data security investigations in 15 minutes or less.

## Getting Started with Security Copilot for Purview

### What You Need to Know First

Security Copilot (SC) provides two experiences for data security:

**Standalone Experience** (securitycopilot.microsoft.com)
- Full-featured copilot interface
- Purview plugin enabled by default for your tenant
- Supports open-ended prompts, prompt suggestions, and promptbooks
- Best for: Exploratory investigation, prompt development, complex multi-step workflows

**Embedded Experience** (Purview portal)
- Copilot button in Purview alerts and user context
- Faster access to data security workflows
- Summarize alerts, investigate users, get risk summaries
- Best for: Quick triage, alert context, inline investigation

### Enable the Purview Plugin (First Time Only)

1. Go to [securitycopilot.microsoft.com](https://securitycopilot.microsoft.com)
2. Bottom left: **Sources** > **Plugins**
3. Find **Purview** and toggle **On**
4. You'll see confirmation: "Purview plugin enabled"

This allows SC to access:
- Your DLP alerts and policies
- IRM cases and user activities
- Data classification and sensitivity labels
- Policy insights

### Review System Capabilities

Before using prompts, understand what SC can do:

1. In SC, click the **sparkle icon** (top right of prompt bar)
2. Select **See all system capabilities**
3. You'll see the full list of available Purview operations:
   - Get Data Risk Summary
   - Get User Risk Summary
   - Summarize Purview Alert
   - Triage Purview Alerts
   - Zoom Into Purview Data Risk
   - Zoom Into Purview User Risk

These are the building blocks that all prompts in this framework use. If a prompt references a capability that's not listed, it may be in preview or not available in your region.

## Using Prompts from This Repository

### How to Copy and Use a Prompt

1. **Navigate to the relevant prompt book** (e.g., [prompts/dlp/prompt-book.md](prompts/dlp/prompt-book.md))
2. **Find the prompt** you need (they're organized by scenario)
3. **Copy the entire prompt text**
4. **Paste into Security Copilot** at securitycopilot.microsoft.com
5. **Replace parameters** (see below) with real values
6. **Submit and monitor** the response

Parameters appear as angle brackets: `<user>`, `<alertId>`, `<timeframe>`, etc.

### Parameterized Prompts

Prompts in this framework use **angle-bracket parameters** for flexibility:

Example prompt:
```
Show me all DLP alerts involving user <user@contoso.com> from the past <30> days
```

Replace with your values:
```
Show me all DLP alerts involving user sarah.chen@contoso.com from the past 30 days
```

Common parameters:
- `<user@contoso.com>` or `<user>` = actual user email
- `<alertId>` = alert GUID from Purview (visible in alert details)
- `<timeframe>` = number of days (7, 30, 90)
- `<filename>` = specific file or document name
- `<policyName>` = DLP or IRM policy name

### Prompt Suggestions vs. Open-Ended Prompts

**Open-Ended Prompts** (in this framework)
- You write the full prompt
- SC understands natural language
- Works well for: custom scenarios, follow-up questions, exploratory investigation
- Example: "Show me the top five high severity DLP alerts"

**Prompt Suggestions** (built into SC)
- Appear as clickable buttons in SC sidebar
- Generated by SC based on your context
- Best for: quick common actions
- Note: You can also copy these into the framework for reuse

Use open-ended prompts from this framework, but also pay attention to SC's suggestions—they're often contextual and valuable.

### Tips for Better Results

1. **Be specific**: "Show me DLP alerts from the Legal team in the past 14 days" works better than "Show me alerts"
2. **Use parameters consistently**: Replace all angle brackets before submitting
3. **Start simple**: Try a basic prompt first, then refine based on results
4. **Check the response**: If SC says a capability isn't available, you may be in a region or cloud where that feature is in preview
5. **Follow the prompt chain**: Prompts in playbooks are ordered for a reason—results from one inform the next

## Building Your Own Promptbooks

A promptbook is a saved sequence of prompts that SC executes step-by-step. Promptbooks are parameterized (you fill in variables like user name) and reusable.

### When to Build a Promptbook

- You have a workflow you repeat regularly (e.g., "investigate new IRM case")
- You want to reduce SCU consumption by pre-optimizing prompt order
- You're preparing a scenario for a customer workshop
- You want analysts on your team to follow the same investigation steps

### Step-by-Step: Create a Promptbook

**Step 1: Build and Test Prompts in Live SC**
1. Go to securitycopilot.microsoft.com
2. Manually enter prompts one at a time
3. Test with real data
4. Note which prompts work, which don't, which order makes sense
5. Keep the working prompts

**Step 2: Create the Promptbook**
1. Click **Promptbooks** (left sidebar in SC)
2. Click **+ New promptbook**
3. Name: descriptive (e.g., "DLP Incident Investigation - Quick")
4. Description: one sentence explaining when to use it
5. Add tags for discoverability (e.g., #dlp #incident #triage)

**Step 3: Add Prompts**
1. Click **+ Add prompt**
2. Paste your first tested prompt
3. If using parameters, wrap them in angle brackets: `<user@contoso.com>`
4. Click **+ Add another prompt**
5. Repeat for all prompts in your workflow

**Step 4: Set Parameters**
1. SC will auto-detect parameters in angle brackets
2. For each parameter, set a friendly label (e.g., "User Email Address", "Alert ID")
3. Optionally set a default value (optional)

**Step 5: Set Sharing & Test**
1. **Visibility**: Private (just you) or Shared with your team
2. Click **Create promptbook**
3. Click **Test promptbook** to verify parameter substitution
4. Run it with real data and validate results

**Step 6: Refine**
1. If a prompt doesn't work, edit it in the promptbook
2. If ordering isn't optimal, reorder prompts
3. Save changes
4. Test again

### Best Practices for Promptbooks

- **Keep them focused**: A promptbook should have a clear goal (3-7 prompts is ideal)
- **Order matters**: Each prompt should logically flow from the previous one
- **Parameterize wisely**: Only expose parameters users will actually change (don't over-parameterize)
- **Test with edge cases**: What happens if a prompt returns no results? Is the next prompt still valid?
- **Document assumptions**: Add a description explaining prerequisites and expected use
- **Monitor SCU cost**: Each prompt costs SCUs; long promptbooks consume more credits

### Sharing Your Promptbook

Once tested:
1. Share the promptbook (via SC sharing settings)
2. Document it in your team's wiki or share via Teams
3. Provide guidance on what parameters to change
4. Include the use case and success criteria

Example documentation:
```
PROMPTBOOK: DLP High-Risk Alert Investigation
WHEN TO USE: When a high-severity DLP alert lands and you need to quickly assess risk
PARAMETERS TO CHANGE: <alertId>, <user@contoso.com>
EXPECTED TIME: 5 minutes
SCUS: Approximately 4-5 SCUs
SUCCESS: You have determined if the alert is a true positive and what file/user is involved
```

## Using SC Agents in Purview

Agents are autonomous SC workflows that handle a specific task. You set them up once, then they run automatically or on-demand.

### Available Agents in Purview

**DLP Triage Agent**
- Automatically categorizes DLP alerts
- Requires: Active-mode DLP policies (not simulation)
- Output: "Needs attention" / "Less urgent" / "Not categorized"
- Best for: Large alert volumes, consistent classification

**IRM Triage Agent**
- Automatically categorizes IRM alerts and cases
- Output: "Needs attention" / "Less urgent" / "Not categorized"
- Best for: IRM case priority management

**DSPM Posture Agent**
- Natural language data discovery and classification
- Automatically discovers sensitive data patterns
- Output: Risk assessment and remediation suggestions
- Best for: Data posture assessments, governance workflows

**Data Security Investigations Agent**
- Credential scanning and anomaly detection
- Output: Credential exposure findings, behavioral anomalies
- Best for: Proactive threat hunting, insider risk investigation

### Set Up a DLP Triage Agent

1. **In Purview portal**, go to **Data Loss Prevention** > **Alerts**
2. Click **Agents** (top right)
3. Click **+ Create new agent**
4. **Name**: e.g., "DLP Alert Triage - High Severity"
5. **Type**: Select **DLP Triage Agent**
6. **Configuration**:
   - Alert scope: Filter which alerts the agent processes (e.g., "High severity only", "Past 7 days")
   - Alert limit: How many alerts to process per run (default: 50)
   - Schedule: Manual trigger or automated (e.g., daily at 9 AM)
7. **Custom instructions** (optional): Add context for your organization
   - Example: "Categorize alerts involving external recipients as 'Needs attention'"
   - Example: "Categorize alerts from test file shares as 'Less urgent'"
8. Click **Create agent**

The agent will then run on schedule and triage alerts automatically. Results appear in Purview as tags on each alert.

### Set Up an IRM Triage Agent

1. **In Purview portal**, go to **Insider Risk Management** > **Cases**
2. Click **Agents** (if available in your preview access)
3. Follow the same steps as DLP Triage Agent
4. Configure for IRM-specific categories (e.g., "Exfiltration" vs. "Policy violation")

### Custom Instructions for Agents

Agents use custom instructions to make smarter decisions. Good custom instructions:

- Explain your organization's context
- Define what "Needs attention" means for your business
- Give examples of categorization logic
- Reference your policies or compliance requirements

Example:
```
Our organization classifies alerts as "Needs attention" if:
1. The recipient is external to the tenant
2. The file contains Healthcare data (sensitivity label: HIPAA)
3. The user is in a high-risk department (Finance, HR, Legal)
4. This is the user's third alert in 30 days

Otherwise, classify as "Less urgent" unless the alert is policy-violating or involves multiple recipients.
```

## Daily Workflows by Role

### For Analysts: Morning Triage

1. **Log into SC**: securitycopilot.microsoft.com
2. **Check DLP alerts**: Use [prompts/dlp/prompt-book.md](prompts/dlp/prompt-book.md) - "Show me top unreviewed alerts"
3. **Check IRM alerts**: Use [prompts/irm/prompt-book.md](prompts/irm/prompt-book.md) - "Show me new insider risk cases"
4. **Quick categorize**: Use your configured Triage Agents to batch-categorize routine alerts
5. **Deep dive high-risk**: Pick the top 2-3 alerts using the DLP/IRM investigation playbooks

Total time: 10-15 minutes to triage, then deep dive as needed.

### For Analysts: Incident Investigation

1. **Get the alert ID** from Purview or your SIEM
2. **Open the relevant playbook**: [playbooks/dlp-incident-investigation.md](playbooks/dlp-incident-investigation.md) or [playbooks/irm-case-triage.md](playbooks/irm-case-triage.md)
3. **Follow the steps**: Each step is a prompt you submit to SC
4. **Document findings** in your investigation template
5. **Determine action**: Is this a false positive, user training, or escalation?
6. **Record the decision** and mark the alert as reviewed in Purview

Expected time: 20-30 minutes per incident (varies by complexity).

### For CSAs: Customer Engagement

**Pre-Engagement (1-2 weeks before)**
1. Review [prompts/workshop/prompt-book.md](prompts/workshop/prompt-book.md)
2. Select 3-4 safe demo scenarios
3. Test each scenario in a staging tenant
4. Create a "demo script" with expected SC responses
5. Prepare talking points on capabilities and limitations

**During Demo**
1. Use the workshop promptbook or safe demo prompts
2. Show real (or realistic) data from SC
3. Demonstrate triage agents in action
4. Honestly explain what SC cannot do
5. Gather feedback on use cases most relevant to customer

**Post-Engagement**
1. Use [prompts/executive/prompt-book.md](prompts/executive/prompt-book.md) to generate a summary report
2. Share relevant promptbooks with the customer
3. Provide links to this framework (if customer is internal)
4. Identify follow-up: Pilot? Full deployment? Custom agent build?

### For Admins: Operational Hygiene

1. **Weekly**: Use [prompts/operations/prompt-book.md](prompts/operations/prompt-book.md) to review policy violations and remediation
2. **Monthly**: Review agent categorization accuracy (do the "Needs attention" alerts match your expectations?)
3. **Quarterly**: Use [prompts/executive/prompt-book.md](prompts/executive/prompt-book.md) to generate compliance reports
4. **Continuous**: Feed learnings back into custom agent instructions

## Building Custom SC Agents

If built-in agents don't meet your needs, you can build a custom SC agent. This is advanced; see [agents/README.md](agents/README.md) for full guidance. Quick overview:

### Three Ways to Build Custom Agents

**1. NL2Agent (Easiest)**
- Go to SC Agent Builder
- Describe in natural language: "Triage alerts involving external file shares"
- SC auto-generates the agent workflow
- Best for: Quick, simple agents

**2. Agent Builder Form**
- Fill out a structured form with triggers, actions, and logic
- Define: When to trigger, what to do, how to categorize/report output
- Best for: Standard business logic

**3. YAML Manifest (Advanced)**
- Upload a YAML file defining agent behavior
- Full control over prompt sequence, parameters, logic
- Best for: Complex multi-step workflows or integration with custom APIs

See [agents/README.md](agents/README.md) for detailed instructions and examples.

## Known Limitations and Gotchas

### Real Limitations (Not Bugs, Just How SC Works)

**DLP Triage Agent**
- Only works with **active mode** DLP policies (not simulation mode)
- Processes files up to 2 MB in size
- Alert content >30 days before agent enablement is out of scope

**IRM Triage Agent**
- Only analyzes SharePoint file content in preview (not email or device logs yet)
- Requires IRM+ or IRM+ Risk investigation license

**SC Context & Performance**
- Prompt responses can vary (SC is a generative model, not deterministic)
- Large promptbooks may return inconsistent results if ordering changes
- SCU consumption is real; prompt optimization matters for cost

**Geographic & Compliance**
- SC is not available in sovereign clouds (China, DoD)
- GCC/GCC-High (US government) only for some features; check your region
- Commercial clouds are GA; check support status in your region

**No Write-Back**
- SC cannot directly remediate in Purview (no delete, no apply policy)
- SC output informs your decisions; you execute actions in Purview
- This is intentional for audit and control

### Common Gotchas

**"The prompt is asking for a capability I don't see"**
- The feature may be in preview in your region
- Enable the Purview plugin (Sources > Plugins)
- Check if you have the required license (SC requires licensing)

**"The agent is categorizing alerts wrong"**
- Tune custom instructions (be more specific about your logic)
- Test the agent with a subset of alerts first, then expand
- Remember: Agents learn from patterns; give them clear examples

**"My prompt results are inconsistent"**
- Resubmit the same prompt twice—SC results can vary
- Add more specificity to your prompt (e.g., exact user email, not just "user")
- Check that all parameters are correctly substituted

**"The DLP Triage Agent doesn't see my alerts"**
- Make sure your DLP policies are in **active mode** (not simulation)
- Verify the agent's alert scope filter matches your alerts
- Check agent status in Purview—is it running?

**"SCU consumption is higher than expected"**
- Long promptbooks cost more SCUs (each prompt = SCU cost)
- Optimize: Can you combine some prompts into one?
- Use Triage Agents for bulk categorization (cheaper than interactive prompts)

### Honesty About What SC Isn't

SC is **not**:
- A replacement for deep forensic investigation tools (use Activity Explorer, Audit Log Search)
- Able to write policies or remediate directly
- Capable of analyzing data outside Purview's scope (email bodies, on-premises data)
- A machine learning model that learns from your corrections (each session starts fresh)
- Suitable for automated, high-confidence categorization without human review

SC **is**:
- A powerful co-investigator that accelerates human analysis
- Able to contextualize alerts with user behavior, risk patterns, and policy insights
- Effective at explaining what happened and why a DLP/IRM alert fired
- Useful for bulk triage when combined with agents and custom instructions
- Best when used with human judgment, especially for high-stakes decisions

## Troubleshooting

### SC says "Purview plugin is not available"
1. Check: Is Purview plugin enabled? (Sources > Plugins > Purview toggle)
2. Check: Is your tenant in a supported region? (SC is commercial clouds only, GA)
3. Check: Do you have SC licensing assigned?
4. Wait 5-10 minutes for plugin enablement to propagate, then retry

### A prompt says "Please enable X capability"
1. Open SC system capabilities (sparkle icon > See all)
2. Search for that capability
3. If not listed, it's in preview or not available in your region
4. Check SC release notes or your CSA for regional availability

### Triage Agent isn't processing alerts
1. Check: Is the agent enabled in Purview? (Agents section)
2. Check: Do alerts match the scope filter? (e.g., if you filtered "High severity only", it won't process Medium alerts)
3. Check: Are the alerts >30 days old? (Agents only see recent alerts)
4. Check: For DLP, are your policies in active mode? (Not simulation)
5. Wait 5-10 minutes for the agent to process and retry

### Promptbook results seem wrong
1. Double-check all parameter substitutions (e.g., user email is correct)
2. Try submitting the first prompt manually to verify it works
3. Check if the alert/user/file still exists (real-time data may have changed)
4. Rerun the promptbook (SC results can vary)

### SCU consumption is higher than expected
1. Reduce promptbook length (fewer prompts = fewer SCUs)
2. Use Triage Agents for bulk work (cheaper per-alert than interactive prompts)
3. Optimize prompts: Can you ask for less data or be more specific?
4. Monitor your SC usage dashboard (SC admin center) to see consumption trends

## Quick Reference: Prompt Types

| Type | Best For | Example |
|------|----------|---------|
| **Open-Ended** | Exploratory investigation, custom scenarios | "Show me all DLP alerts from the Sales team involving external recipients" |
| **Guided Follow-Up** | Refining based on SC response | "Tell me more about the user's risk profile" |
| **Parameterized** | Reusable workflows, hand-off to team members | "Investigate user <user@contoso.com> for exfiltration in past <30> days" |
| **Promptbook** | Multi-step workflows, consistent methodology | "DLP Incident Investigation" promptbook (6 sequential prompts) |
| **Agent** | Automation, bulk processing, scheduled runs | DLP Triage Agent running daily on new alerts |
